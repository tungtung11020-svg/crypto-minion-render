# Архитектура и модель угроз

## Архитектура
Один репозиторий содержит Telegram long-polling бот, FastAPI API, асинхронный слой SQLAlchemy, SQLite/Alembic, сервисы оплаты и лицензирования и независимый PyQt6-клиент. Бизнес-логика вынесена из транспорта в `services`, поэтому SQLite можно заменить PostgreSQL изменением URL и миграций. `start.py` запускает бот и API в одном процессе; компоненты также запускаются отдельно.

## Границы доверия
- Telegram подтверждает `successful_payment`; сообщения и скриншоты пользователя не доверяются.
- Desktop-клиент не содержит pepper/приватный ключ и считается недоверенным.
- Сервер подписывает короткоживущий offline token Ed25519; клиент проверяет публичным ключом.
- Полный ключ существует только в момент выдачи. База хранит HMAC и последние 4 символа.

## Угрозы и меры
| Угроза | Мера | Остаточный риск |
|---|---|---|
| Перебор ключей | 160 бит энтропии, SQLite rate limit, единая ошибка, аудит | Распределённые атаки требуют reverse proxy/WAF |
| Повтор платежа | UNIQUE payment ID/order, транзакция, идемпотентность | Сбой Telegram требует сверки журнала |
| Кража БД | HMAC+pepper вне БД | Компрометация БД и `.env` раскрывает поиск ключей |
| Подделка token | Ed25519, приватный ключ только сервер | Компрометация приватного ключа требует ротации |
| Копирование token | device hash + DPAPI | Администратор ПК может модифицировать клиент |
| Манипуляция игровым ledger | уникальный result_id, серверная запись | Полностью локальную игровую статистику нельзя надёжно защитить от владельца ПК |
| Утечка ключа в лог | фильтр и запрет логирования тела | Сторонние crash dump требуют отдельной настройки |
| Потеря SQLite | WAL, backup API, 7 копий | Копии на том же диске не защищают от поломки диска |

API в публичном режиме должен работать только через HTTPS. Для реальной нагрузки добавьте reverse proxy, централизованный rate limit и внешние резервные копии.
